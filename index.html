  • As you might already seen, TVs became connected devices over the last several years too
  • But like everywhere else multiple companies provided multiple technologies to make it a connected device
  • so there were Set Top Boxes, [CLICK] Smart TV Sticks [CLICK] and Game Consoles
  • With every piece of device it enabled your TV to have somekind of connected expierence
  • I am talking about Apps or web surfing capabilities
  • Even though as a customer you have the great choice to pick one of the devices to equip your TV
  • As developer you need to build 3 or more different apps in different languages
  • That is hell
  • HbbTV tries to define one common standard to bring digital applications to the TV
  • So instead of buiding apps for Apple TV, Chromecast or Rokus TV which all have different platforms to build apps
  • With HbbTV you can build one app with web technologies that is running on all TVs natively
  • So what is HbbTV?
  • HbbTV is an ETSI standard
  • like HTML and CSS which is defined by W3C by an standardisation institute
  • ETSI: European Telecommunications Standards Institute
  • Introduces a handful of technical components and is mainly based on existing standards
  • HTML / CSS / JS pages rendered by a browser engine on the TV

Broadcast-independant applications

  • Not assosicated with a broadcast stream
  • TV apps provided by the device vendor

Broadcast-related applications

  • Associated and opened with a specific broadcast stream
  • Automatic or explicit launch by user via Red Button
  • HbbTV = Hybrid Broadcast Broadband => broadcast + broadband stream
  • When customer opens a broadcast channel the TV receives next to the broadcast stream (audio/video signal) a small data package that contains an URL of the HbbTV app
  • AIT - Application Information Table
  • HbbTV complaint TVs can read that table and reads the url from it
  • This loads the App on top of the TV video stream
  • The first version of the standard was introduced in 2010 that allowed to build web apps based in a CE-HTML (XHTML-based standard for designing webpages with remote user interfaces for consumer electronic) that supports:
    • ECMA Script 3 (released 1999)
    • XHTML 1.0 transitional/strict
    • CSS 1.0
    • => just think about that you have to build html apps from last century
  • With HbbTV 1.5 that added some technical components with HbbTV 2.0 (released last year) the spec finally requires to support HTML5 and CSS3 (even though the JS API support is still different across all devices)
  • Within the last 5 years 3 specifications were released that differentiate a lot, unlike desktop browser people don’t update their TVs every other month
  • Device fragmentation is insane
  • Problem: Broadcaster need to support all versions of HbbTV to reach all all people in front of the TV
  • It’s like someone tells you: build Netflix that runs on IE5
  • And make sure it is performant because the CPU on TVs is limited
  • There is also no develop community around HbbTV (No frameworks, No Tooling)
  • Extreme difficult to ensure quality across devices and HbbTV versions
  • Question: So how we can bring HbbTV development up to speed?
  • Don’t reinvent the wheel
  • Selenium / Webdriver have been proven to be successful for test automation
  • TVs provide no native interface to browser internals
  • In order to provide some kind of automated testing functionality we need to go the Selenium way
  • Which is: integrate an automation helper (no SSL issues, all on HTTP, Raspberry)
  • BUT: do it the Webdriver way and use the DevTools protocol to allow web authoring

DevTools Backend

  • Standalone implementation of the Chrome DevTools protocol
  • Proxy to inject automation script
  • Similar what Weinre did back in the days
  • A proxy modifies the HTML response to inject an automation script
  • Initializes TCP connection to a server
  • Emulates remote debugging interface
  • Not everything can be emulated
  • JS API is pretty comprehensive and can do most of the stuff that we need for debugging

HbbTV Page Object


import { Page } from './mpat.page'

class BroadcastProbingSystemPage extends Page {
    open () {
        super.open()
        browser.keys('down').keys('down').keys('down').keys('down').keys('down').keys('enter')
        browser.waitUntil(() => this.activeMenuItem.getText() === 'BPS')
        browser.pause(2000)
    }

    get title () { return 'DVB-T2 Broadcast Probing System' }
    get videoThumbnail () { return $('.video-content .thumbnail') }
    get video () { return $('.video-wrapper > object') }

    isVideoPlaying () {
        return browser.waitUntil(() => {
            return browser.execute(function () {
                return document.querySelector('.video-wrapper > object').playState === 1
            }).value
        }, 10000, 'video never started')
    }
}

export default new BroadcastProbingSystemPage()
export { BroadcastProbingSystemPage as Page }
